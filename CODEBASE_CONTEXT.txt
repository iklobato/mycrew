================================================================================
1. PROJECT OVERVIEW
================================================================================

Mycrew is an AI-powered development team automation system built on CrewAI that 
autonomously implements GitHub issues. It clones a repository, analyzes the issue, 
explores the codebase, creates an implementation plan, writes code, runs tests, 
reviews for security/performance issues, and creates a pull request.

The pipeline follows a sequential crew flow: Issue Analyst → Explorer → (optional TactiqResearch) → Clarify → Architect → Implementer → Test Validator → Reviewer → Commit

Entry points:
- CLI: python -m mycrew.main (or 'mycrew', 'kickoff', 'run_crew' commands from pyproject.toml)
- Webhook server: python -m mycrew.webhook (FastAPI server on port 8000)
- Kickoff client: python -m mycrew.kickoff_client (HTTP client to trigger pipeline)
- Webhook registration: python -m mycrew.register_webhook

Python version: >=3.10,<3.13

Key dependencies (from pyproject.toml):
- crewai[litellm,tools]==1.9.3
- pyyaml>=6.0
- fastapi>=0.104.0
- uvicorn[standard]>=0.24.0
- httpx>=0.25.0,<1.0.0
- pydantic-settings>=2.10.1,<2.11
- requests>=2.28.0

Required environment variables:
- OPENROUTER_API_KEY (required for LLM calls)
- GITHUB_TOKEN (required for cloning repos, creating PRs)

Optional environment variables:
- HUGGINGFACE_API_KEY (alternative LLM provider)
- SERPER_API_KEY (web search)
- TACTIQ_TOKEN (meeting context)
- GITHUB_WEBHOOK_SECRET (webhook verification)
- PROVIDER_TYPE (openrouter or huggingface)
- CODE_PIPELINE_LOG_LEVEL (DEBUG, INFO, WARNING, ERROR)
- CREWAI_TRACING_ENABLED
- HOST, PORT (webhook server)
- REDIS_URL
- DEFAULT_DRY_RUN, DEFAULT_BRANCH

6. DATA FLOW
================================================================================

ENTRY POINT 1: CLI (python -m mycrew.main or kickoff-client)
1. User provides GitHub issue URL or local repo path
2. _main() parses args, configures logging
3. kickoff() creates PipelineState and CodePipelineFlow
4. flow.kickoff() starts sequential pipeline

ENTRY POINT 2: Webhook Server
1. POST /webhook receives JSON payload
2. _handle_github() verifies signature (if configured)
3. _extract_github_params() pulls issue_url from payload
4. _run_kickoff_background() calls kickoff() in background thread
5. Returns 202 Accepted immediately

PIPELINE FLOW (sequential crews):
1. explore() - ExplorerCrew analyzes repo structure
   - Input: issue_url, issue_data
   - Output: exploration_result (tech stack, dependencies, conventions)

2. analyze_issue() - IssueAnalystCrew parses issue
   - Input: issue_url, issue_data, exploration_result
   - Output: issue_data with task, clarifications

3. tactiq_research() (optional) - TactiqResearchCrew
   - Input: task, issue_analysis, exploration, tactiq_meeting_id
   - Output: tactiq_result (sufficient_info flag)

4. clarify() - ClarifyCrew asks human questions
   - Input: issue_url, issue_data, exploration_result
   - Output: issue_data updated with clarifications

5. architect() - ArchitectCrew creates plan
   - Input: issue_url, issue_data, exploration_result
   - Output: architecture_result (file-level plan)

6. implement() - ImplementerCrew writes code
   - Input: issue_url, issue_data, exploration, plan, clarifications, prior_issues
   - Output: implementation_result

7. review() - ReviewerCrew checks implementation
   - Input: issue_url, issue_data, exploration, plan, implementation
   - Output: review_result (APPROVED or ISSUES)

8. validate_tests() - Runs test_command subprocess
   - Input: test_command from state
   - Output: validation_result (passed/failed)

9. commit() - CommitCrew creates PR
   - Input: all prior results, branch, dry_run
   - Output: commit_result (PR URL or "dry run")

RETRY LOOP:
- If review returns ISSUES, retry_implement() increments retry_count
- Loops back to implement() up to max_retries times

OUTPUT:
- If dry_run: changes exist locally, no push/PR
- If normal: branch pushed, PR created via gh CLI
- Callback URL receives completion status (if provided)

8. KNOWN PAIN POINTS
================================================================================

1. PipelineState is a "god model" - Contains 20+ fields for all pipeline stages.
   All state flows through this single class, making it hard to maintain.
   Consider: Stage-specific state classes or more focused state objects.

2. CodePipelineFlow._run_crew does too much - builds inputs, sets context,
   runs crew, handles errors all in one method. Multiple responsibilities.

3. Main.py is very long (677 lines) - The flow orchestration has grown complex
   with many stage methods. Consider extracting stage handlers or using a
   more declarative flow definition.

4. Factory functions in factory.py have long if-elif chains for stage mapping.
   This could be refactored to a dictionary mapping or registry pattern.

5. Tool initialization in PipelineCrewBase creates tools at class definition time
   using get_pipeline_context() which may not be set yet. Potential ordering issues.

6. tactiq_research() in main.py has unreachable code - after "return self.architect"
   there's more code that will never execute (line 354-355).

7. Reviewer verdict parsing is fragile - multiple isinstance checks and string
   matching ("APPROVED" in result_upper). Consider structured output parsing.

8. _make_pr_body in create_pr_tool.py has many string length checks and truncation
   logic that could be centralized.

9. get_llm_for_stage and llm_with_fallback have complex fallback logic that could
   be simplified. The retry with sleep on rate limits is a bit naive.

10. Some crews (ClarifyCrew, TactiqResearchCrew, TestValidatorCrew) override
    the crew() method instead of relying on @crew decorator, inconsistent pattern.

11. RepoShellTool._run is extremely long (200+ lines) with complex output
    truncation and sampling logic. Could be broken into helper methods.

12. Settings uses pydantic BaseSettings but then accesses via get_settings()
    singleton pattern - could be simplified to just use the Settings instance.

13. The _extract_github_repo method in input_builder.py duplicates logic that
    exists in multiple places for extracting github_repo from issue_data.

14. No separation between crew configuration (agents.yaml, tasks.yaml) and
    runtime configuration. YAML files are loaded at import time.

15. Some hardcoded values like webhook URL and timeouts should probably be
    in config files or environment variables for easier deployment changes.
